# CLAUDE.md — grok-desk

Fork of phuryn/grok-build-vscode, rebranded as **Grok Desk** (`contactjayclatty.grok-desk`).
Upstream history and process notes below still apply unless they mention marketplace publisher IDs.

VS Code sidebar extension for **xAI's Grok Build CLI**, driven by `grok agent stdio` over the [Agent Client Protocol (ACP)](https://agentclientprotocol.com). Thin client — all session state, MCP servers, subagents, memory, and plan-mode bookkeeping live in the CLI.

## Status

See [CHANGELOG.md](CHANGELOG.md) for the current version and full release history (the Marketplace/GitHub-published build can lag the local working tree). The whole test suite is **grok-free** — CI never spawns the binary; grok-dependent probes live separately in `research/*.cjs`. Plan mode is **enabled** and enforced client-side (see `research/plan-mode.md` § Resolution). Smoke-tested end-to-end on Linux, Windows-via-WSL, and the **native Windows build** (`irm https://x.ai/cli/install.ps1 | iex`) — `cli-locator` resolves `grok.cmd`/`grok.exe` and `terminal-manager` runs the agent's commands under PowerShell on Windows (`pwsh`→`powershell`, mirroring the standalone CLI; #46), `/bin/sh` elsewhere.

This file is the project map: it describes the **current** architecture, not a per-version log.

**Editing discipline — keep this file lean.** When behavior changes, *rewrite* the affected lines to the new current state; do **not** append a note about what changed or why the old way was wrong. Concretely, none of these belong here:
- version-by-version history or "fixed in 0.2.x / removed in 1.4.x" → [CHANGELOG.md](CHANGELOG.md);
- bug post-mortems / "used to… / this broke because…" narrative → a code comment or `research/*.md`;
- evidence dumps ("one capture: 17.7s", "a real case: 74 messages") → `research/*`.

State the rule, name the function / constant / config-key an agent would grep for, and link the deep-dive — **one fact, one home**. Every feature already has a `research/*.md`; `docs/architecture.md` holds the event-flow detail; [docs/ACP-feedback.md](docs/ACP-feedback.md) is the upstream-facing summary of grok-CLI/ACP friction, agent-family wire differences, and our workarounds (update it when a new workaround lands or an old one becomes obsolete). If a paragraph explains *how we found* a bug rather than *how the code behaves now*, it's in the wrong file.

**Grok CLI compatibility (#22).** A since-fixed Windows-only `grok agent stdio` regression (the agent didn't read stdin until EOF, so ACP startup hung — *"Grok exited (code null)"* / *"ACP request timed out"*) is worked around by pinning Windows to a supported build. The guards live in [src/cli-locator.ts](src/cli-locator.ts) and are all **Windows-only no-ops off win32**: `isStdioBrokenGrokVersion` flags the broken version range, `maybePinBrokenCli` pins those builds to `GROK_STDIO_DOWNGRADE_TARGET` (the single-source-of-truth supported build) before spawning, and `shouldReactivelyDowngrade` → `downgradeBrokenCli` is the evidence-driven backstop that downgrades + retries once on an observed startup failure above the target. When a newer Windows-verified build ships — **verify with the session/new probe, not just initialize** — bump the constant and widen the broken range. Exact versions, diagnosis, and probes: `research/stdio-eof-regression.md`; timeline: [CHANGELOG.md](CHANGELOG.md).

## Session pool (Agent Dashboard)

The sidebar runs a **pool of live sessions** (one `grok agent stdio` process each) with one `focused` (the one rendered). Each session's state is a `Session` (`src/session.ts`); the sidebar holds `focused` + a `Set<Session>` `pool`. **Re-focus is lossless**: every session buffers the webview posts that built its chat (`Session.buffer`), so `focusSession` is `clearMessages` + replay of that buffer — no grok reload, no kill, even mid-turn/mid-approval. Clicking a *cold* session (no live process — reaped or pre-window) loads it from disk into a fresh member (`openSession`); a live one re-focuses instantly. Two guards keep it safe: a **per-session `gen`** (bumped only when *that* session's client is torn down, captured by handlers when wired) so a backgrounded session's in-flight events aren't judged stale when focus moves; and **session-scoped `emit(session,…)`** (buffers + forwards only when focused) vs UI-wide `post(…)` (status dots, sessions list). **Status dots.** Every history row shows a dot whose color is a *derived* value (the pure `computeDot` in `src/session-pool.ts`), not the raw `status`: **gray** at rest (idle / read / cold / loaded-from-disk all collapse to one gray — the user never learns warm-vs-cold), **blue** working, **yellow** needs-you, **green** finished-and-unread, **red** errored-and-unread. The green/red is an **unread badge**, not a live state: a turn that ends while the session is *not* focused sets a persisted `unread`/`unreadError` flag in `SessionMetaOverride` (globalState); focusing/opening the session clears it (`markRead`). So the badge **survives reaping AND reload** (it's meta-backed, decoupled from the live process) — that's the payoff: come back and the green dots are exactly the sessions with results you haven't seen. No timer/decay — a never-opened session stays green (it *is* still unread). `computeDot({liveStatus, unread, unreadError})` precedence: working → needs-you → unread(error?) → none. Dots are pushed one-at-a-time via `pushDot`→`sessionDot` (cheap, no disk read) plus a full `dots` map (`dotForId` over disk entries + pool) on each list refresh; palette in CSS — Working=`--vscode-charts-blue`, Needs-you=`--vscode-charts-yellow`, unread=`--vscode-charts-green`, error=`--vscode-errorForeground`, none/default=`--vscode-descriptionForeground`. **Reaping** is the pure `selectReapable` (`src/session-pool.ts`): an **idle TTL** (untouched ≥1h, swept every 5 min) + an **LRU cap** (~8 live; least-recently-used eligible evicted past it), but it **never** reaps the focused or a `working`/`needs-you` session (so the cap can be exceeded when everything spare is busy, by design). Reaping is silent — kill the process, recompute the dot (an unread reaped session **stays green**; a read one goes gray); re-click reloads from disk. Recency (`Session.lastActiveAt`) is stamped on create/focus/made-busy via `touch`. **CLI-update guard:** the explicit *Update Grok Build CLI* action (`updateGrokCliOnDemand`) confirms before tearing the pool down if any session is `working`/`needs-you`; the silent startup auto-update has nothing in flight by construction, so it doesn't ask. Per-workspace scope. The impure `voice-recorder`/`voice-streamer` (ffmpeg spawn + STT fetch/WebSocket) are smoke-tested manually via the `research/voice-*.cjs` probes (`voice-stt-probe`, `voice-e2e-verify`, `voice-stream-probe`, `voice-stream-verify`, `voice-cost-probe`).

**Empty-primer-session cleanup (#24).** Every time the extension opens it primes a fresh grok session (`ensurePrimed`); if the user never sends a real message that leaves an empty, primer-only session on disk, titled from grok's primer-derived summary ("… Primer v4 Plan Mode …"). Two mechanisms keep those out of history. **(1) Recycle on leave (in-memory, bulletproof):** `parkFocused` knows a session is primer-only via `hasHistory === false` (the primer is sent through `ensurePrimed`, *not* the `send` path, so it never sets `hasHistory`), and on New Session / switching away it deletes that session's on-disk dir (`removeSessionFromDisk`), not just its process — so at most one untitled **New session** ever exists, and `postSessionsList` forces its display name to "New session" while live (in-memory `!hasHistory`), never grok's primer title. **(2) Startup sweep (legacy cleanup, content-confirmed):** `sweepEmptyPrimerSessions` runs once per activation over the newest `SWEEP_SCAN_LIMIT` by mtime; it skips live + renamed (`customName`) sessions, then reads `chat_history.jsonl` and deletes only if `isEmptyPrimerSession` confirms the session got our primer and **zero real user queries** — so a real session (even one with a primer-contaminated title) or a session we didn't start (no primer) is never touched. **The chat-history content check is authoritative regardless of `num_messages`** — `EMPTY_PRIMER_MAX_MESSAGES` only gates the no-content title-heuristic fallback (an agentic primer turn can balloon the count, so it must not veto the content read). Detection is content-based and agent-agnostic: grok keys the role on `type` (not `role`), wraps prompts in `<user_query>` **but not always** (slash commands like `/imagine` arrive UNWRAPPED), and tags injected `<system-reminder>`/context turns `synthetic_reason`; `extractUserQueries` handles all of that, so the same logic works for the `grok-build`/`grok-build-plan` and `cursor` (composer) agents alike.

## History pagination

The history popover loads sessions **one page at a time** (`SESSION_PAGE_SIZE = 100`, newest-first) so it stays responsive with thousands of sessions on disk. The cost of the old behavior (read + `JSON.parse` *every* `summary.json` on every open, then render every row) grew linearly and stalled the popover at scale.

**Two pure primitives** replace the monolithic full-scan `listSessions` (which is kept only for callers that genuinely need the whole list):

- `indexSessions({fs, grokHome, cwd})` — one `stat` per session dir, **no reads/parses**, returning every id ordered newest-first by `summary.json` **mtime**. mtime is the cheap last-activity proxy: grok rewrites that file (it holds `updated_at`) on every turn. We sort by mtime here *because the id is a UUIDv7 whose timestamp is creation, not last activity* — id-sort would be wrong.
- `readSessionEntries({fs, grokHome, cwd, ids, overrides})` — reads + parses `summary.json` for **exactly the given ids** (the visible page) and applies `customName` overrides. The exact `updated_at` order is re-applied within the loaded page (mtime is only an approximate sort key).

**Host orchestration** (`postSessionsList` in `sidebar.ts`): `indexSessions` orders everything cheaply; a `mtimeById` map drives an **mtime-keyed read cache** (`sessionCache`, `readEntriesCached`) so a re-open / load-more / search reads only entries whose `summary.json` actually changed — steady-state opens cost ~zero reads. It posts `{type:"sessions", entries, activeId, dots, offset, total, hasMore, query}`. **Search** is server-side and complete: a non-empty `query` warms the whole catalog once (cache-backed) and filters by display name across **all** sessions, not just the loaded page. The cache is invalidated on `rename` (changes displayName without touching mtime), `delete`, and `clearAll`. **Live-session synthesis:** a brand-new session has no `summary.json` yet, so on the **first, unfiltered page** `postSessionsList` injects a synthesized `liveSessionEntry` (pinned newest-first) for any live `pool` member whose `activeSessionId` isn't on disk, named from the best in-memory source (generated/renamed `customName` → first user message → `"New session"`); the next refresh replaces it once grok flushes the file. `hasMore` is computed from the on-disk page *before* injection (load-more pages are disk-only), and these ids are never on disk so they can't duplicate onto a later page.

**Webview** (`media/chat.js`): `offset === 0` replaces the list, `offset > 0` appends (de-duped by id); a scroll-near-bottom listener requests the next page (`hasMore`/`sessionLoading` guards keep it to one request per boundary); the search box posts a **debounced** (180ms) `listSessions` with the query; a host-driven refresh that arrives unfiltered while a search is active triggers a **sticky-search** re-request so the filtered view isn't clobbered. The "Clear all" footer shows when any non-active session exists (loaded *or* on a later page, via `total`). **Popover positioning:** `positionDropdownPopover` **right-aligns** the popover to the panel edge (`right: 6px`/`left: auto`) and grows it leftward — width-independent, no measurement. It **caps min/max width to the panel** (`min(360|280, parentWidth − 2·EDGE)`, overriding the CSS `min-width: 280px`/`max-width: 360px`) so a long session name **ellipsizes** in a narrow panel instead of overflowing. **Resize/hide:** a `window` `resize` listener re-fits it live while open; a `document` `visibilitychange` listener `closePopovers()` when the view is hidden (a resize while hidden fires no event, so close rather than reappear stale).

**Perf simulation** (`npm run test:perf`, `test/sessions.perf.ts` via `vitest.perf.config.ts`) — opt-in, out of `npm test`/CI. At N=5000 it asserts the op counts: first open drops file reads from **5000 → 100** (~98%), steady-state re-open is **0 reads** (mtime cache), search warms once then **0 reads**, with a modeled-latency projection (~3250ms → ~310ms) and a real in-memory parse-cost wall-clock (~2.7x, disk-latency-free lower bound).

### Test taxonomy — three layers

There are **three** kinds of tests, and it matters which is which:

1. **`npm test` — grok-free unit/DOM/integration suite (1004 tests).** Pure logic, happy-dom tests that drive the real `media/chat.js`, a real-`/bin/sh` TerminalManager smoke, and a fake-CLI ACP integration suite (`test/fixtures/fake-grok-acp.cjs`). **Never spawns the real `grok` binary.** Runs in a few seconds with no network, no login, no subscription. This is the floor — every change keeps it green. (Two separate opt-in suites live outside `npm test`: `npm run test:perf` runs the session-history perf simulation — `test/*.perf.ts` via `vitest.perf.config.ts` — and `npm run test:integration` runs the `@vscode/test-electron` smoke below; both are matched only by their own config so they stay out of `npm test`.)
2. **CI — the *same* suite, plus a non-blocking Electron smoke.** `.github/workflows/ci.yml`'s `test` job runs `npm ci && npm run compile && npm test && npm run package` on a clean Ubuntu box — **CI's `test` job ≡ layer 1, verbatim** (no `grok` binary, no auth, no subscription, so it *cannot* touch the real CLI — the whole reason layer 1 is grok-free). A second job, `integration`, runs the `@vscode/test-electron` smoke under `xvfb` (`npm run test:integration` — activate the extension, assert commands registered, resolve the webview via the *missing-CLI* onboarding path so it needs no grok). It's a **required gate** (validated passing against a real VS Code Extension Host before landing; if a Linux/xvfb quirk ever surfaces, fix it rather than demoting the gate). The smoke is still grok-free — it exercises host glue (activation, `getHtml`/CSP, `localResourceRoots`, command registration) the unit suite structurally can't reach. Compiles in isolation (`integration/tsconfig.json` → `out-integration/`), so it's out of `npm test` and the extension build.
3. **`npm run test:live` — on-demand pre-release suite against REAL grok (`scripts/live-tests.cjs`).** Spawns the actual `grok agent stdio` and exercises the surfaces layers 1–2 can't: the real ACP handshake, a **capability-drift probe** (`capabilities` — snapshots `promptCapabilities` and hard-asserts the `image:false` baseline, so the day grok flips it the gate FAILs with a "the vision workaround may be removable" message; paired with `vision-prompt` which pins that vision *actually* works, this is an advertised-vs-actual drift detector), a prompt round-trip, a **mid-turn cancel** (`cancel-mid-turn` — the Stop-button contract: `session/cancel` as an id-less notification must settle the in-flight prompt with a cancelled stopReason and leave the session usable, #37), **mid-turn steering** (`interject` — the #52 Steer contract: `_x.ai/interject` must reach the MODEL mid-turn *and* leave the turn uncancelled; an unadvertised RPC, so this is the drift canary that tells us if Steer silently degraded to queueing), **conversation forking** (`session-fork` — #48: a new id, the parent's history carried, loadable; pins `ForkSessionRequest`'s required param shape), **concurrent sessions** (`parallel-sessions` — two `grok agent stdio` processes on one workspace, mirroring the Agent Dashboard pool: overlapping prompts complete independently with no cross-talk), session restore, **plan-mode enforcement modeled as the two real flows** (primer → plan → `[Plan rejected]` keeps the gate up & asserts 0 workspace mutations *and* byte-identical seed files — the containment canary — → `[Plan approved]` drops the gate & asserts implementation can land; the old single-turn test invented an impossible approve+no-verdict+gate-up state that pushed grok-4.5 into force-writing), and the v1.4.x generative features. It **reuses the real compiled modules** (`out/acp-dispatch.js`, `out/plan-gate.js`, `out/grok-primer.js`, `media/webview-helpers.js`) — genuine wire output through the same `isMediaGenToolCall`/`extractGeneratedMediaPaths`/`isSubagentToolCall`/`shouldBlockWrite`/`GROK_PRIMER` the extension uses. **The `release.*` scripts now run it by default** (mandatory gate; `-SkipLive`/`--skip-live` is the explicit escape hatch), and you should still run it by hand before a release. It needs a logged-in grok + subscription and burns credits, so it's never in `npm test` or CI. Flags: `--smoke` (fastest lane — handshake + capability-drift only, no prompt turns), `--quick` (skip the slow tests incl. the now-4-turn plan-mode), `--only=`, `--skip=`, `GROK_BIN=…`. A SKIP does not fail the gate — only a FAIL does. Real-grok **diagnostic probes** (`research/*.cjs`) remain manual one-offs; the live suite is the repeatable gate.

**So:** local `npm test` == CI's `test` job (both grok-free); CI additionally runs the non-blocking grok-free Electron smoke (mirror it locally with `npm run test:integration`). The real-grok tests are a separate, mandatory pre-release gate — always run before a tag/release (no need to ask), never on every commit.

### grok CLI version + updating

The native-Windows build (`irm https://x.ai/cli/install.ps1 | iex`) installs from the **stable** channel. The extension pins Windows to the supported build (`GROK_STDIO_DOWNGRADE_TARGET` in [src/cli-locator.ts](src/cli-locator.ts); see § Status → Grok CLI compatibility). **grok does not auto-update.** Updating is the explicit `grok update` command: `--check [--json]` checks without installing, bare `grok update` installs the latest on the current channel, `--stable` (default, weekly) / `--alpha` switch channels, `--version <X.Y.Z>` pins a build. `grok update` can only rename the binary when no grok process holds it open (else *"cannot rename locked executable"*), so the extension's update path tears the whole pool down and **awaits each process's actual exit** before updating; on Windows it `taskkill /T /F`-kills the process **tree** (grok backgrounds children that keep the binary locked) and retries once on a lingering lock (`isLockedBinaryError`). Re-run `npm run test:live` after any CLI update — the wire format is the thing that drifts.

## ACP surfaces implemented

- `initialize` → `session/new` / `session/load` → `session/set_model` → `session/prompt` lifecycle
- **Model switching is agent-aware.** Models belong to *agent types* — `grok-build`/`grok-build-plan` vs the `cursor` agent that owns the Composer models. The CLI binds the agent at spawn and locks it after the first turn (incl. our primer), so a live `session/set_model` only works within the same agent; a cross-agent switch errors `MODEL_SWITCH_INCOMPATIBLE_AGENT` ("Start a new session"). `switchModel` in `sidebar.ts` tries the live switch, and on that error (detected by the pure `isIncompatibleAgentError` in `acp-dispatch.ts`) persists the pick to `grok.defaultModel` and restarts — `newSession` re-applies the model *before* the primer runs, while the agent is still rebindable. No history → transparent restart; with history → the same Summarize/Just-Restart prompt as an effort change (shared `pickRestartMode`/`restartSession` helpers). **Empty-session cleanup:** a restart on a *primer-only* session (no real conversation — the common case when you flip models/effort a few times right after opening) gets the no-prompt transparent path *and* `discardRestartedEmptySession` deletes the abandoned grok session dir afterward, so repeated switches don't pile up identical primer-only sessions in history; the pure `carrySessionName` (in `sessions.ts`) moves any user rename (`customName`) from the discarded id onto the fresh one so the chosen name survives. The same cleanup runs on the effort-change empty-session branch (guarded so a dead client on a session *with* history keeps its history). The toolbar model label resolves IDs to user-facing names via the pure `modelDisplayName` helper. **Versioned-id handling:** grok's `set_model` echoes a *versioned* id (`grok-build` → `grok-build-0.1`) that isn't in `availableModels` and carries no name/context of its own, so `setModel` anchors to the *requested* list id (always resolvable to a name + window); the pure `resolveModelId` in `acp-dispatch.ts` (exact → longest base-id prefix → input) is the fallback for grok-provided ids on `newSession`/`loadSession`. Context window is per-model (`_meta.totalContextTokens`: grok-build 512K, Composer 200K), recomputed in `chat.js` on every `modelChanged`.

- Sessions: list/resume via `session/load` (grok stores them at `~/.grok/sessions/<urlencoded-cwd>/<id>/`); rename/delete metadata in `context.globalState["grok.sessionMeta"]`. Clear-all is the relief valve for an overgrown store; the pagination is the steady-state fix.

On resume the question replays as a `tool_call` (questions in `rawInput`) + completed `tool_call_update` (answer text); chat.js suppresses the generic tool chip and rebuilds a read-only "You answered" card from that replay. - `session/set_mode` wired; the picker exposes **Agent**, **Plan**, and **Auto accept** (the auto-approve mode — internally still the `"yolo"` mode id / `autoApprove` flag; "YOLO" survives only in the picker description). The CLI's non-plan mode id is `"default"` (not `"agent"`), captured as `ACT_MODE_ID` in `sidebar.ts`. **Config-forced auto-approve (#31):** grok's global `permission_mode = "always-approve"` (config.toml, set via Shift+Tab or `/always-approve`) auto-approves every session server-side and is invisible over ACP, so `newSession` reads the config (`configForcesAlwaysApprove` in `grok-config.ts`) and starts the session in "Auto accept" with a one-time notice, instead of a misleading "Agent"; the extension can't disable a global config setting per-session.
- **Plan mode is enforced client-side** (mirror of YOLO). The CLI's `x.ai/exit_plan_mode` still treats any client response — result *or* error — as approval (re-verified broken in 0.2.3), so we don't rely on it. Instead `src/plan-gate.ts` gates the two *mandatory* server→client choke points: `fs/write_text_file` (block writes resolving inside the workspace cwd) and `terminal/create` (block anything not on the read-only allowlist). grok's own `~/.grok/sessions/<…>/plan.md` write lands *outside* the workspace and is allowed (and snooped to recover the plan text — `exit_plan_mode` arrives with `planContent: null`). Approve → drop the gate + send an "implement it now" follow-up prompt; Keep planning → gate stays up; a comment-less Cancel is silent — the notice is the whole UX, the `[Plan cancelled]` follow-up runs as a hidden turn (no dots, no ack bubble), and turn end always clears the dots indicator (an empty ack used to orphan it). Every verdict also cancels + content-suppresses the remainder of the planning turn (`suppressPlanReject`) — responding to `exit_plan_mode` unblocks it server-side and grok's primer-trained continuation is contentless "I'll wait for your verdict" filler that grok doesn't persist to history, so it must not paint live either (Approve skips the reject path's `agentReset` so pre-card narration stays on screen). A verdict also buffers a `planResolved` event (the plan twin of `permissionResolved`) so a warm re-focus replays the card collapsed instead of resurrecting the buttons. Resolved cards (live, replayed, or disk-restored) show just the plan-file link + verdict label — no inline plan text; the Show/Hide toggle survives only when no plan file exists (snapshot creation failed / legacy). Entering plan mode *any* way (incl. agent-initiated `current_mode_update: plan`) raises the gate; it's lowered only by explicit user action, never auto-lowered by CLI mode flapping. The verdict protocol is taught by a hidden **primer** (`src/grok-primer.ts`) — it tells grok to ignore the bogus `exit_plan_mode` result and read `[Plan approved]`/`[Plan rejected]`/`[Plan cancelled]` (optionally + a comment) from the follow-up message. **The primer fires eagerly + non-blocking** as its own hidden turn the moment a session goes live — on new AND restored sessions, and after `/compact` — *not* in front of the user's first prompt and *not* at composer-disabling cost (`ensurePrimed` in `sidebar.ts`, kicked off from `startSession`'s success path). It returns a reused `session.primingPromise` (`src/session.ts`) so a racing first send awaits the *same* in-flight primer (grok runs one turn at a time) and is released the instant the silent primer acks; the user can type and send straight away — their bubble shows immediately (a "Grokking…" indicator stands in until grok produces content) while the real prompt is held behind the primer in code. On restore the primer is *re-sent* on first go-live rather than trusted from replayed history (a `/compact` can drop it). The primer's silent turn is hidden by a session-level `suppressContent` flag checked in `post`/`emit` against `SUPPRESS_TYPES` (which deliberately **excludes** `userMessage`/`agentStart` so a user send that races the background primer still paints its own bubble + Grokking indicator). The primer (`PRIMER_VERSION`/`PRIMER_MARKER`) is deliberately minimal — the plan-verdict protocol plus an explicit *do not use any tools / do not read files / do not search the workspace / do not take any action* constraint ending in "Reply with exactly: ok" (grok-build is agentic; anything beyond the protocol tempts a slow pre-turn workspace exploration). The pure `isPrimerText()` (version-agnostic `PRIMER_PATTERN`) detects the primer when grok replays it so the bubble is hidden + not counted toward plan positions. Full course: `research/understanding-plan-mode.md`.

- `_meta.totalTokens` → context donut. The host strips a `totalTokens: 0` report before it reaches the webview (`gateZeroTokenMeta` in `acp-dispatch.ts`) — grok reports 0 for `/session-info` (context untouched) and `/compact` (context shrunk, not emptied), so 0 is never a real measurement (#39); the donut keeps its last real value and the "Compacted." bubble is the compact worked-signal. The real count for the cases the meta can't cover rides the `contextUsage` host message, from two sources: **(a) cold restore** — the pure `readContextUsage` (`sessions.ts`) reads the session's persisted `signals.json` (`contextTokensUsed`/`contextWindowTokens`), else the donut sits at 0 until the first turn; **(b) post-/compact** — the CLI recomputes `signals.json` only when the NEXT inference turn ends, never at the compact turn's own end (`research/signals-refresh-probe.cjs`). **Primary:** the fresh count rides the live `_x.ai/session_notification` rail — `auto_compact_completed.tokens_after` (the pure `contextUsedFromCompactNotification` in `acp-dispatch.ts`, handled in the `xaiNotification` listener), which fires for BOTH a manual `/compact` and automatic (context-full) compaction and sets `Session.sawCompactNotification`. **Fallback (pre-rail CLIs, e.g. the Windows downgrade target 0.2.72):** when that notification didn't arrive, `refreshContextAfterCompact` (`sidebar.ts`) runs a hidden CLI-local `/session-info` turn and `parseSessionInfoContext` (`acp-dispatch.ts`) pulls the exact count from its reply ("**Context:** N / M tokens"). The notice is a dedicated message (not a `messageChunk`) so it finalizes any active bubble and can't reorder the agent's answer
- **Steer — mid-turn injection (#52).** `_x.ai/interject` (`{sessionId, text}` → `{status:"queued"}`) queues into the session's pending-interjection buffer, drained at the agent's next safe point. It **does not cancel** the turn and loses no in-flight tool work — that is the whole difference from Stop-and-resend, and it's pinned by the `interject` live test (the model must obey mid-stream *and* the turn must not end cancelled). **`grok.steerByDefault`** (webview-only, default **false**; gear → Config & debug → *Steer by default*, hidden when the CLI can't interject) is the policy: off = today's queue (with an on-demand Steer button), on = a send-while-busy **skips the queue** entirely. It branches in `queueOutgoing` (chat.js) — the single choke point both typed sends and dictated utterances pass through — and falls back to queueing on all three nothing-to-steer-into cases (`busyLocked` priming, `!steerSupported`, not busy). A queued block gains a **Steer** button while `body.turn-busy`; clicking it `dequeueSend`s + posts `steerSend`, and `steerSend` (`sidebar.ts`) paints the bubble optimistically then interjects. Steering carries **plain text only** — it bypasses `prompt-builder`, so no context envelope, no chips, no `/command` dispatch. On -32601 the host emits `steerUnavailable` (latches the button off), **re-queues the text**, and warns — a capability gap must never eat a message.
- **Fork — branch the conversation (#48).** Gear → Session → *Fork conversation* → `forkSession` → `_x.ai/session/fork`. Params are `ForkSessionRequest`'s three **required** camelCase fields (`sourceSessionId`/`sourceCwd`/`newCwd`; `{sessionId}` alone is -32602); the response's `newSessionId` is handed to `openSession`, so the fork loads like any history row. It copies session files under `grok_home()` **only** — the workspace is untouched, so **code is not rewound** (`/rewind` + `rewind_points.jsonl` is grok's separate file-snapshot feature). The parent is verifiably unchanged. Forks carry `session_kind: "fork"`, and `buildEntry` (`sessions.ts`) only special-cases `"subagent"`, so they appear in history with no client change. **Naming:** the fork is stamped with the parent's history name behind a **leading** `(Fork)` tag via the pure, **idempotent** `forkDisplayName` (`sessions.ts`) — leading because history rows ellipsize at the panel edge, so a trailing tag is the first thing to vanish; forking a fork stays `(Fork) Foo`, never stacks. The parent name comes from `sessionDisplayName` (`sidebar.ts`), which mirrors the list's precedence (`customName` → `session_summary` → first message) **except that it rejects a primer-derived summary** (`isPrimerSummary`): we prime every session, so grok titles it from the hidden primer and `session_summary` is routinely "… Primer v4 Plan Mode …" — inheriting that would propagate an internal name for an invisible message through every fork-of-a-fork. The parent name is resolved **before** the fork RPC (a turn landing mid-fork rewrites `summary.json` and grok's generated title with it) and written as the fork's `customName` **before** `openSession`, so no surface flashes grok's own title; the toast then points the user at the untouched original in history. **Whole-session only, deliberately:** `targetPromptIndex` works on the wire but truncates `chat_history` *without* truncating `updates.jsonl`, so a fork-at-message-N would replay a conversation the model has forgotten — blocked upstream (`research/grok-build-oss-findings.md` § 3a).
- **`_meta.usage` → billing breakdown (#53).** Nested inside the prompt result's `_meta` (and mirrored on the rail's `turn_completed.usage`), extracted by the pure `extractPromptUsage`. **Three quantities, never conflated:** `usage.*` is the WHOLE prompt's billing (across `modelCalls`); the flat `inputTokens`/`outputTokens` siblings are the LAST model call only; `totalTokens` is CONTEXT size (one probed turn: 16371 context vs 32488 billed). They don't decompose into each other, so the donut arc stays context-only and the popover shows usage as its own sections. **No cache-*creation* field exists anywhere in the CLI** — it is omitted, never faked as 0. `addUsage` sums the session total (absent + absent stays absent), gated by `usageIsRealMeasurement`: a `/compact` turn replays the previous turn's siblings verbatim, so counting it would double-bill. The popover shows **Model calls**, without which billed input (which routinely dwarfs context, since every model call re-sends the conversation) reads as a bug. The total is **ours** — grok reports per-prompt only and `signals.json` persists just context size — so it's persisted to `SessionMetaOverride.usage` and re-seeded by `restoreUsage` on load; a session with no stored total shows context only.
- **Vision input (image paste/upload).** Ctrl+V in the composer, drag-drop, and the + picker attach raster images (png/jpe?g/gif/webp — SVG deliberately stays a *path* chip so grok can read/edit the source; 20MiB cap per the xAI docs) as inline `{type:"image", mimeType, data:<base64>}` content blocks in `session/prompt`, plus a `[Image #N]` tag line in the text block (`envelope\n\ntext\n\ntags` — text keeps position 0 so `/commands` still dispatch; disk imports carry the origin path in the tag, `[Image #1] (assets/x.png)`, so grok can act on the real file). **The CLI still advertises `promptCapabilities.image:false` but actually accepts the blocks** (verified 0.2.87; behavior over advertisement — no capability gate, the `vision-prompt` live test pins it instead; see `research/vision-input.md` + `research/vision-probe.cjs`). Pending images live in a session-NEUTRAL staging dir (`<globalStorage>/image-staging/` — a grok session dir gets deleted by the empty-session cleanup, which would kill a pending screenshot), are pre-read + validated at send (any unreadable image **blocks the send** with chips intact — a dangling tag sends the model hunting for an image it never got), and are unlinked after send / chip removal, with an age-gated sweep on activation for orphans. `[Image #N]` numbering is session-scoped (`Session.imageCounter`, re-seeded from replayed prompts on restore). The webview holds send while a paste's FileReader is in flight (`pendingPaste`) and keeps the text half of a mixed clipboard; chips are host-owned — `send` carries only the text.
- **Command details (#41).** The extension runs every shell command itself (the mandatory `terminal/*` handlers), so each command row carries an always-available, collapsed-by-default expandable detail — a Claude-Code-style **IN/OUT block** on the shared code-chip surface: IN = the FULL command text from `rawInput.command` (grok truncates its titles; scrolls past ~140px), OUT = the complete captured output (scrolls past ~260px) — snapshotted at `terminal/release` in `acp.ts` (`commandDone` → `commandOutput` host message; the buffer is exactly what grok received — same byte cap — plus a 100K display cap in `sidebar.ts`). Every tool row is ONE line (`.tool-item-label` span, CSS-ellipsized at the row edge as backstop; `toolLabel`'s 40-char clamp is the deliberate scannable-summary trim — the full text lives in the detail; a COMMAND row instead reads "Run \<program\>" via the pure `commandProgramLabel` (webview-helpers.js): first token, path-stripped/de-quoted, + one non-flag **bare** subcommand (a *quoted* next token, or one containing a path separator, is an argument/banner not a subcommand, so it's dropped — `Write-Output '=== 1. git status ==='` → "Write-Output"; `node research/x.cjs` → "node"), first-statement-only, "command" fallback. A **`(cd dir ; cmd)` subshell** (grok's POSIX navigate-then-run idiom, emitted even against PowerShell) is unwrapped — the `( )` stripped and a leading `cd <dir>` prelude skipped — so it labels the real command (`(cd … ; node x)` → "node"), not "(cd"; the cd-skip is scoped to subshells, so a user-typed `cd src && npm test` still reads "cd src" — so `git status`/`npm test`/`Get-Date` stay scannable while a `node -e "…"` payload collapses to `node`); a lone RUNNING command's group is expandable immediately (`cmd-single` — header chevron shows mid-run and expanding auto-opens the row's IN/OUT; multi-tool groups keep their chevron until the batch closes); expandable command rows carry a TRAILING chevron (› collapsed ↔ v expanded, `expanded` class) + hover brightening (`has-details`, `wireCommandToggle` — plain rows like the output poller don't react), and clicks inside the block never collapse it (text selection). Exit codes merge into the output: exit 0 with output is silent, exit 0 with **no** output renders a muted `✓ done · no output` marker (no empty `(no output)` pre), non-zero renders an `[Error] exit N` marker + error-tinted output **and rolls the failure up to the row (`.tool-failed`) + group (`.has-error`)** — a failed command signals at a glance like a `status:"failed"` tool (`markToolFailed`), not just inside its OUT box, a kill renders muted `[Cancelled]`. Output attaches by **two paths, one per agent execution model**: **grok-build** delegates commands via `terminal/create`, so the client snapshots the buffer at release and attaches by exact command string (`commandOutput` → `state.pendingCommandDetails`, `attachCommandOutput` in chat.js) with a standalone fallback row so output is never dropped; the **cursor/Composer** agent runs commands in its own CLI-side persistent shell (never `terminal/create`), so its output rides the **completed `tool_call_update.rawOutput`** and is attached by **`toolCallId`** (`maybeAttachToolResultOutput` + the pure `extractToolResultOutput` in webview-helpers.js — order-independent, because Composer runs commands in parallel and finishes them out of issue order, so a FIFO/order-based guess would misattribute). Both paths reference the same `details` node stored on the `pendingCommandDetails` entry (keyed by both `command` and `toolCallId`), which survives a lone command's flatten to `.tool-flat` (the flatten MOVES the chevron + details nodes). `grok.expandCommandOutputs` (webview-only, default false; gear → Config & debug → **Expand tool details** — label renamed to match the two Command-Palette commands, setting KEY unchanged; mirrors the showThinking pattern via `applyExpandCommandOutputs`) pre-opens every detail **and** every detail-bearing tool GROUP (a group is detail-bearing iff it has a `.has-details` row — a command's IN/OUT **or** an edit's diff; read/explore-only groups stay collapsed) — the group open state is applied both live (`applyExpandCommandOutputs`) and at group finalization (`closeToolGroup`) — except on a group the **user** has toggled (`el._userToggled`, set by the header click), which `closeToolGroup` leaves alone: expanding a *running* batch to watch it must not be undone the instant it finishes. An explicit setting change or Expand/Collapse-All latch flip still force-applies through `applyExpandCommandOutputs`, so a global action always wins. On-demand, the Command Palette `grok.expandAllToolDetails` / `grok.collapseAllToolDetails` (titled *…(This Session)*) → `sidebar.setAllToolDetails` **`emit`s** (not `post`s) a `setAllToolDetails` host message that flips a per-session **latch** — `state.toolExpandOverride` (tri-state `null | true | false`) in chat.js. The latch force-expands/collapses **every** group (incl. explore/edit) + IN/OUT box — including a still-running batch — and keeps applying to later tool calls, because all render points flow through the effective-expand helpers `groupShouldExpand`/`detailShouldExpand` (`override ?? persisted-setting`). It beats the persisted setting (last action wins; flipping the gear switch or the `expandCommandOutputs` message clears it), **rides the session replay buffer** (the `emit`, not a `post`) so it survives Agent-Dashboard focus-swaps, and resets on a cold reopen (`resetForNewSession`) — never persisted to disk. CSS shows a running group's chevron once it's `.expanded`. Failed non-shell tools surface their reason via `toolFailureText`, which mines variant-keyed `rawOutput` blobs (e.g. `list_dir` → `NotFound`, `read_file` → `FileReadError`) after `message`/`content`/`error`, else the generic "Tool call failed." Radius scale: bubble 12 → block code/IN-OUT 8 → inline chip 6 (`--chat-radius-*`). Live-session only: the CLI doesn't replay terminals on `session/load`.
- **Edit diffs inline (#45).** Every `kind:"edit"` tool row carries an always-visible `+N −M` change count (green/red, rolled up onto the collapsed group header as **path-deduped** totals, e.g. *Edited 2 files · +7 −2*) **plus** an expandable inline diff on the same detail surface commands use — so an edit is auditable in chat even under **Auto-accept**, where no permission card fires. This is decoupled from the permission flow on purpose: the diff `content` block (`type:"diff"`, with `oldText`/`newText` = the replaced region) rides the `tool_call_update` regardless of permission mode, and whether a permission card even appears is grok-build/platform-dependent, not a config toggle (verified 0.2.99: default config sends 0 permission requests for in-workspace edits; `research/edit-diff.md`). The pure `computeLineDiff(oldText,newText)` (`webview-helpers.js` — LCS backtrack, CRLF-normalized for compare+display, empty region = 0 lines so a new file reads as pure adds) yields the `+/−` counts and the diff rows. **A block is expanded into one hunk per replaced SITE** (`extractDiffSites` in chat.js): the block's `oldText`/`newText` is the search *pattern*, so a `replace_all` block describes only the token and alone renders 148 occurrences as one meaningless `+1 −1` — `_meta.details[]` is the complete account (one entry per site, `{old_string, old_line, new_string, new_line, context_before, context_after, line_prefix}`, real 1-based **post-edit** file lines; verified 12/12 against ground truth, `research/edit-diff-lines.log`). Per delivery shape: **completed + session/load** carry `details[]` → a site each, rendered `line_prefix + old_string` → `line_prefix + new_string` (there is **no `line_suffix`** on the wire, so the line's tail is genuinely unavailable and is not reconstructed); the **echo** carries no `details[]`, only a first-site-only block `_meta.{old_line,new_line}` → one approximate hunk the completed update then upgrades; neither (a whole-file Write's `_meta:{}`) → one hunk seeded 1 via `fileLineOr1` (also the guard for absent/bogus — 0/negative/non-number). A creation (`old_string:""`) takes no prefix, so a new file stays pure adds instead of the prefix becoming a phantom del. `buildInlineDiffRegion(hunks)` renders them **Codex-style**: **ONE `.tool-diff-region` per BLOCK holding N hunks** (never one region per site — the region is a 320px scroll box, so 148 would stack 148 scrollers) of `.tdl` rows, each a `[+/− sign][line-number gutter][code]` grid with a colored left-border stripe + subtle per-line tint; each hunk seeds its own gutter, and a quiet `.tdl-sep` hairline parts **non-contiguous** hunks (never before the first, never between contiguous ones). `MAX_INLINE_DIFF_LINES`=400 is a budget **across** a block's hunks (1000 sites must not paint 2000 rows) with `open diff →` for the remainder — but the `+N −M` is summed over **every** site *before* the cap, so a capped render still reports the true magnitude. Del shows the old-side number, add/context the new-side. The gutter track sizes per region to the widest number rendered (`--tdl-num-w` = `max(4ch, digits+1)`) — 4ch through 999, wider only for a 1000+ line file. The `+/−` glyph is a color-blind affordance beside the border. Colors are **Codex's exact palette** (`--tdiff-*` CSS vars, dark default + `body.vscode-light` override): add `#40c977`/`#00a240`, del `#fa423e`/`#ba2623`, backgrounds the same at 23%/15% alpha — reused for the `+N −M` stat too. `applyToolDiffs`/`attachDiffPreviewToToolItem` handle BOTH the live `tool_call_update` and the session-load completed `tool_call` replay shapes (#30); a single call can carry multiple `diff` blocks (rendered as an array), each getting exactly **one** `open diff →` (per-site would emit 148 buttons; its payload stays the block's own `oldText`/`newText`). **grok reports each edit's diff twice** — an optimistic pre-write echo, then the authoritative completed update — and for a whole-file Write that OVERWRITES a file the echo carries `oldText:""` while the completed one carries the real prior content, so the echo must not win: idempotency is keyed on `item._diffSig` (the serialized diffs, **sites included**, so the `details[]`-seeded render outranks the echo's approximate one), making an identical replay a no-op while a *different* later diff repaints the row (reusing the detail node, or `wireCommandToggle`'s listener would strand + double-bind). **Totals paint as each edit lands**, not at batch close — the data is on the wire ~2.2–3.3s before the turn ends (`research/edit-diff-timing-probe.cjs`). `recomputeGroupDiffTotals` *recomputes* from the rows' `_diffStat` (so a corrected row replaces rather than double-counts) and `paintGroupDiffTotals` owns a `.tool-group-diff-totals` span, re-painted after each of the two things that wipe the header: `addToToolGroup`'s per-call `hdr.innerHTML` rebuild and `closeToolGroup`'s label reset. Painting earlier still, from the initial `tool_call`'s `rawInput`, is deliberately NOT done: it buys ~30ms and reconstructs the same wrong `oldText:""` for a Write. None of this expands anything — totals ride the *collapsed* header; only `grok.expandCommandOutputs` / the latch open rows. The native `open diff →` link (→ `sidebar.ts` diff editor) stays for the full side-by-side. `summarizeTools` counts edits by unique `toolFilePath` so "Edited N files" matches the deduped header totals.
- **Generated media (v1.4.x).** `/imagine` (`image_gen`, or `image_edit` for reference-photo edits) and `/imagine-video` (`video_gen`; older/Linux builds `image_to_video`) are subscription-only and do **not** return ACP image blocks — grok writes the file into its session dir (`images/*.jpg`, `videos/*.mp4`) and reports the path as a JSON string in the completed tool result's `text` content. - **Subagent rows.** A delegation is a tool call whose `_meta["x.ai/tool"].name` is `spawn_subagent` (grok-build) or `Task` (the Composer agent) — that marker is **authoritative both ways** for the pure classifier (`isSubagentToolCall`, `webview-helpers.js`): grok titles Grep/Read calls with their query/filename, so a grep FOR "spawn_subagent" is titled exactly that and title matching false-cards; without `_meta`, exact tool names + `rawInput.subagent_type` + the legacy background heuristics apply (the `get_command_or_subagent_output` poller stays excluded). The restore-created card is marked `subagentReplayed` and excluded from **live** lifecycle matching (mis-attribution guard) but NOT during replay (`state.replaying || !replayed`), so it still consumes its own re-forwarded lifecycle. FIFO spawn-tagging + child-id matching keep the right card. A history replay settles never-completed rows at replay end. Real wire capture: `test/fixtures/composer-subagent-session.jsonl` (replayed end-to-end by `test/subagent-replay.dom.test.ts`); deep-dive: `research/subagents.md`.

## Cross-platform notes

- `terminal-manager.ts` selects the shell via `resolveTerminalShell(platform, resolve, pref)`: Windows runs the agent's commands under PowerShell (`pwsh.exe`→`powershell.exe`, else cmd.exe via `shell:true`) to mirror the standalone CLI (#46); everywhere else `shell:true` → `/bin/sh`. Node runs a string shell as `<shell> -c "<command>"` (pwsh + Windows PowerShell both accept `-c`, and Node passes the command string through faithfully — embedded double quotes / JSON / paths-with-spaces survive; only cmd/bash-style `\"` escaping doesn't, which grok won't emit for a PowerShell target). `whichOnPath` skips the `\WindowsApps\` Store-alias stub. `grok.terminalShell` (`"auto"`|`"cmd"`, host-read → `setTerminalShellPreference`, cache-clearing) is the escape hatch back to cmd. Don't hardcode shell paths. **The agent is told which dialect to WRITE for** via `GROK_SHELL` in its spawn env (`buildEnv` in `sidebar.ts` sets it from the pure `grokShellEnvValue(resolvedTerminalShell(), platform)` — `pwsh`/`powershell`/`cmd` on Windows, unset on POSIX, and never overriding a user-set value): grok's ACP mode otherwise describes the shell from its own host detection and can emit POSIX idioms for a PowerShell host (§2.9; the `Shell:` line in the model's first message tracks `GROK_SHELL`, `research/oss-surfaces-probe.cjs`). **The `pwsh`-vs-`powershell.exe`-fallback gap is real: Windows PowerShell 5.1 collapses any non-zero native exit to 1 (pwsh 7 preserves it) and rejects `&&`/`||` chains (pwsh 7 accepts them) — install pwsh 7 for full parity.** A fresh shell spawns per command, so the profile re-runs each time (matches standalone's reachable functions, but adds latency and any profile stdout can prepend to captured output; on a `Restricted` execution policy 5.1 skips the profile + warns). The agent only acts on zero-vs-non-zero, which is preserved. Root cause + probe evidence: `research/powershell-terminal.md`.
- `cli-locator.ts` reads `HOME` / `USERPROFILE` env vars first (testability), falls back to `os.homedir()`. Uses `where` on Windows, `command -v` elsewhere. Checks `.cmd`/`.exe`/`.bat` extensions on Windows.
- Tests use `node -e "..."` everywhere, so commands are deterministic across platforms — don't add `pwd`, `awk`, `sleep`, `true`, etc.

## Publishing

**Release procedure — ALWAYS tag + create a GitHub Release (with the `.vsix` attached) on a release push to `main`** (standing convention; mirrors the `v1.0.0…` tag history + GitHub Releases):

**The whole procedure below (steps 2–5) is scripted** — after bumping the version + writing the changelog section (step 1, user-initiated), just run:

```bash
pwsh scripts\release.ps1        # Windows (native) — what we use here
./scripts/release.sh            # macOS / Linux / WSL
```

It reads the version from `package.json`, runs the gate, builds the vsix, commits the working tree (`-MessageFile`/`-Message` override the default `Release vX.Y.Z`), pushes `main`, creates the annotated tag, and runs `gh release create` **with the vsix attached** — extracting the matching `## X.Y.Z` changelog section as the release notes. `-DryRun`/`--dry-run` previews; `-NoTest`/`--no-test` skips the gate. It refuses to run off `main` or when the tag already exists (i.e. the version wasn't bumped). The script does **not** call `vsce`/`ovsx`, and there is **no publish workflow** (`.github/workflows/` is just `ci.yml`) — so the GitHub Release does **not** auto-publish to any store. Store publishing is split by who runs it (step 6): the agent **always publishes Open VSX right after the script succeeds** (part of every release, no separate ask); the **VS Code Marketplace** is manual and Paweł-only.

What the script encodes, step by step:

1. Bump `version` in `package.json` (user-initiated) and add the dated section to `CHANGELOG.md`. As part of this pass, **review the release-facing docs for staleness against what actually ships** — this review is a required step, not optional:
   - `README.md` — features / screenshots / install matrix.
   - `TESTS.md` — test plan vs. what actually ships.
   - `CLAUDE.md` — the project map (§ Module map, § ACP surfaces, § Known limits, the test-count floor): does it still describe the current architecture? It deliberately **doesn't** hardcode the version (it points to the changelog), so it needs no version edit — but its prose can go stale as behavior changes.
   - `docs/architecture.md` — the architecture doc: keep it in sync with new modules / event flows / surfaces.
   - `package.json` itself — description, `engines`, `categories`, `contributes`.

   Keep `CHANGELOG.md` entries **terse** (1–3 bullets per release under Added/Changed/Fixed; deep detail belongs in commits / `research/`, not the changelog); releases before the current minor live in `docs/CHANGELOG-ARCHIVE.md` (linked from the foot of `CHANGELOG.md`).
2. `npm test` (966-test floor, all green) + `tsc -p . --noEmit` clean, **and `npm run test:live` against real grok — mandatory, run without asking**. The `release.*` scripts **now run `test:live` by default** as part of the gate (`-SkipLive`/`--skip-live` is the explicit escape hatch); running it by hand first is still wise since the scripts also commit/tag/push once it's green.
3. Commit + push to `main` (direct-to-main, no feature branches).
4. **Annotated git tag** `vX.Y.Z` at the release commit → `git tag -a vX.Y.Z -m "Release vX.Y.Z"` → `git push origin vX.Y.Z`.
5. **GitHub Release** for that tag → `gh release create vX.Y.Z --title "Release vX.Y.Z" --notes-file <notes> <vsix>` (notes = the new changelog section(s); include any earlier version that was bumped but never released). **Always attach the built `grok-desk-X.Y.Z.vsix` as a release asset** so the exact installable build is downloadable from the release.
6. **Store publish — split by store.**
   - **Open VSX / Marketplace:** optional for this fork — only if a `contactjayclatty` publisher is set up. Prefer GitHub Releases with the `.vsix` attached.
   - **Never announce a store as "live now."** A successful `ovsx publish` is NOT public availability — Open VSX scans new versions before listing them. In issue comments / user-facing messages, say **"~10 minutes"** for BOTH stores (Open VSX and the VS Code Marketplace alike).
   Publishing ≠ tagging either way.

Don't skip the tag/release (or the vsix asset) on a release push. (A pure mid-dev version bump that isn't a release — e.g. the unreleased v1.3.0 voice iteration — is the only exception.)

## Repo conventions

- **Work on `main`, synced to `origin/main`.** `main` is the only working line — start every task from `main` fast-forwarded to `origin/main`. The `dev` branch is **abandoned** (stale far behind `main`); do **not** develop on it. If the working tree is on `dev`, or `main` is behind `origin/main`, switch to `main` and `git reset --hard origin/main` before doing anything else (the published version lives on `main` — a stale checkout will look many versions behind and misnumber the changelog).
- Direct-to-`main`, no feature branches
- **Never commit or push automatically.** Changes reach `main`/`origin` only on the user's **explicit** request (rare) or as part of a release. Otherwise: build + reinstall locally for testing and leave the working tree uncommitted.
- Commits explain the *why*, not the *what*
- Don't introduce abstractions speculatively
- Don't add comments that explain what well-named code already says
- **Keep `CLAUDE.md` a current-state map** (see the editing-discipline note under § Status): rewrite changed behavior in place, never append changelog narrative or bug post-mortems here — link `research/*.md` / CHANGELOG instead
- 1004 tests is the floor — every PR should keep that green. All tests are grok-free (no binary spawn); grok-dependent probes live in `research/*.cjs` and are run manually, never by `npm test` or CI
- **Rebuilding clears older `.vsix` first** — `npm run package` (and the install/release scripts) delete stale `*.vsix` before building, so only the current version is on disk.
